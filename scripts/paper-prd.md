# /paper 스킬 PRD

## 목적

어떤 PDF 논문을 넣어도 일정한 품질의 한국어 블로그 포스트가 나오는 반복 가능한 파이프라인.
논문 하나를 잘 정리하는 것이 목표가 아니라, **구조 자체가 황금알을 낳는 거위**가 되는 것.

---

## 파이프라인 흐름

```
PDF 입력
  → PyMuPDF: Figure 추출 (300×200px 이상, 최대 6개) → assets/ 저장
  → pdfplumber: 텍스트 추출 (최대 100,000자)
  → prompt_template.txt + 기존 카테고리/태그 주입
  → Gemini API (gemini-2.0-flash): 한국어 포스트 생성
  → _posts/YYYY-MM-DD-slug.md 저장
  → git add → commit → push
  → 원본 PDF 자동 삭제 (로컬 용량 절약, --keep-pdf 로 보존 가능)
```

---

## 포스트 구조 (고정 6섹션)

| 섹션 | 항목 수 | 비중 | 역할 |
|------|---------|------|------|
| 1. 연구의 목적 | 2개 | 간결 | 왜 이 연구가 필요한가 |
| 2. 연구의 방법 | 2개 | 간결 | 어떤 방법론, 무엇을 분석했는가 |
| 3. 주요 발견 | 3개 이상 | **본론** | 프레임워크·모델 상세 + 핵심 발견 |
| 4. 결론 및 시사점 | 3개 이상 | 충분 | 현장·설계에 주는 함의 |
| 5. 리뷰어 ADD(+) One | 3개 | 충분 | 독자적 관점 (주목할 지점 / 인접 분야 연결 / 발전 아이디어) |
| 6. 추가 탐구 질문 | 3개 | 간결 | 미탐구 영역, 다른 맥락, 윤리·기술 질문 |

섹션 3·4·5가 전체의 70% 이상을 차지해야 한다.

---

## 섹션 3 설계 원칙

논문이 프레임워크·모델·이론을 제안하는 경우:
- 정의 → 구성요소(각각 별도 항목) → 기존 이론과의 차이 순으로 전개
- 각 구성요소 항목: 이름 + 왜 필요한지 + 어떻게 작동하는지 + 논문 예시
- 이름만 나열하거나 요약으로 때우면 안 됨

실증 연구 논문의 경우:
- 주요 변수, 측정 도구, 이론적 배경을 먼저 정의
- 발견을 예상과 다른 점 중심으로 전개

---

## 형식 규칙

- **번호 체계**: 최상위는 `(1)(2)(3)...`만 사용. 하위 목록은 반드시 `-` 불릿 (2칸 들여쓰기)
- **번호 중첩 금지**: `(1)` 안에서 다시 `(1)(2)` 쓰는 것 금지
- **따옴표 금지**: `'` `"` 불필요한 인용 부호 사용 금지
- **문체**: `~함`, `~됨`, `~임`, `~있음` 단정체. `~합니다`, `~됩니다` 존칭 금지
- **볼드**: 핵심 개념·고유명사·수치만
- **AI 슬롭 금지**: "함께 살펴보자", "주목할 필요가 있다", "흥미롭게도" 등 금지
- **출처**: APA 7판, URL/DOI 미포함 (Gemini 생성 링크 신뢰 불가)

## 파일 구조

```
scripts/
  pdf_to_post.py       # 메인 스크립트
  prompt_template.txt  # Gemini 프롬프트 (섹션 구조·형식 규칙 정의)
  requirements.txt     # pdfplumber, PyMuPDF, google-generativeai 등
.env                   # GEMINI_API_KEY (gitignore)
.claude/commands/paper.md  # /paper 슬래시 커맨드 정의
```

---

## CLI 옵션

```bash
python scripts/pdf_to_post.py <PDF경로>            # 변환 + git push + PDF 자동 삭제
python scripts/pdf_to_post.py <PDF경로> --dry-run  # 출력만, 저장 안 함
python scripts/pdf_to_post.py <PDF경로> --no-push  # 로컬 저장만 (PDF는 삭제됨)
python scripts/pdf_to_post.py <PDF경로> --keep-pdf # 원본 PDF 보존
python scripts/pdf_to_post.py <PDF경로> --model gemini-2.0-flash
```

`_papers/*.pdf`는 `.gitignore`에 등록되어 저장소에 커밋되지 않는다. 기본 동작은 처리 후 원본 삭제 — 로컬 용량이 쌓이지 않도록 한다.

## 품질 기준

- 섹션 3에서 논문의 핵심 기여(프레임워크, 모델, 개념)가 구성요소 단위로 상세히 설명되는가
- 섹션 5에서 논문 내용 재요약 없이 독자적 관점이 전개되는가
- 중첩 번호 체계 없이 `(1)` + `-` 구조로 일관되는가
- 단정체 문체가 전편에 유지되는가
- 출처에 URL/DOI가 없는가

---

## 알려진 제약

- 스캔 PDF (이미지 기반) 텍스트 추출 불가
- Gemini가 `date:` 연도를 임의로 바꾸는 버그 → 스크립트가 생성 후 강제 복원
- 100,000자 초과 논문은 앞부분만 사용 (대부분 학술 논문은 범위 내)
- Windows cp949 인코딩 환경에서 subprocess 출력 깨짐 (기능에는 영향 없음)
