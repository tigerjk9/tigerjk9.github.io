## 살아있는 문서의 탄생
OpenWiki의 핵심은 **살아있는 문서**를 만드는 능력이다. 기존의 위키는 개발자가 직접 수동으로 업데이트해야 했다. 이 과정에서 코드와 문서 간의 불일치가 발생하기 쉬웠다. OpenWiki는 이러한 문제를 근본적으로 해결한다. 코드가 변경될 때마다 문서를 자동으로 갱신하여, 항상 소스 코드의 최신 상태를 반영한다. 문서는 마치 빌드 프로세스처럼 소스 코드에서 파생되어 지속적으로 다시 만들어진다.

## 소스 기반 자동화의 이점
OpenWiki는 특정 GitHub 레포지토리를 입력하면 해당 프로젝트의 문서를 생성한다. 단순히 한 번 만드는 것을 넘어, 코드 변경 사항에 따라 문서를 지속적으로 업데이트한다. 이는 마치 소프트웨어 빌드처럼 문서가 항상 **최신 상태**를 유지하게 한다. 이러한 접근 방식은 문서가 낡아버리는 문제를 구조적으로 막는다.

OpenWiki는 다음과 같은 GitHub 프로젝트들을 문서화한다. 이들 프로젝트는 다양한 분야와 규모를 아우른다.

| 레포지토리 | 설명 | 스타 수 |
|---|---|---|
| **react/react** | 웹 및 네이티브 사용자 인터페이스 라이브러리 | 246k |
| **microsoft/vscode** | 비주얼 스튜디오 코드 | 187k |
| **vercel/eve** | 에이전트 구축 프레임워크 | 3.2k |
| **tailwindlabs/tailwindcss** | 빠른 UI 개발을 위한 유틸리티 우선 CSS 프레임워크 | 95.7k |
| **supabase/supabase** | Postgres 개발 플랫폼 | 105k |
| **rust-lang/rust** | 안정적이고 효율적인 소프트웨어 구축 | 114k |

위 예시들은 OpenWiki가 유명 프로젝트부터 기반 프레임워크까지 광범위한 GitHub 레포지토리의 문서를 효율적으로 관리함을 보여준다.

## Vercel eve와 에이전트 프레임워크
OpenWiki는 **Vercel**의 **eve** 프레임워크를 기반으로 구축된다. eve는 '에이전트를 위한 넥스트JS(Next.js)'로 불리며, **내구성 있는 에이전트(durable agents)**를 구축하는 데 특화된 플랫폼이다. OpenWiki는 이러한 eve의 **쇼케이스** 역할을 한다.

이 시스템에서 웹 애플리케이션은 **내구성 있는 에이전트**를 소비하는 클라이언트 역할을 한다. 코드 분석 및 **위키 생성**은 백그라운드 작업으로 실행된다. 이때 **Neon**에 상태 정보를 저장하고 **Blob** 저장소에 최종 산출물(생성된 위키 문서)을 보관한다. 이 방식은 에이전트가 데이터에서 파생된 결과물을 생성하고 이를 최신 상태로 유지하는 **애플리케이션의 표준 형태**를 제시한다.

## 자동 문서의 한계와 사람의 역할
OpenWiki와 같은 자동화된 문서화 도구는 코드의 **작동 방식**을 설명하는 데 뛰어나다. 즉, 코드가 '무엇을 하는지'에 대한 정보는 정확하고 풍부하게 제공한다. 그러나 코드의 **설계 의도**, 특정 결정의 **트레이드오프**, 그리고 '왜 그렇게 만들었는지'에 대한 맥락은 파악하기 어렵다. 이러한 부분은 여전히 개발자의 통찰력과 경험이 필요한 영역이다. 따라서 OpenWiki는 개발자의 수동적인 문서화 작업을 보완하지만, 완전하게 대체하지는 않는다.
