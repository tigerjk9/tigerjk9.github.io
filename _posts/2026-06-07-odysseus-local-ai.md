## 오디세우스, 나의 AI 작업 공간을 위한 항해

'**오디세우스(Odysseus)**'는 챗GPT나 클로드 같은 대규모 언어 모델(LLM) 기반의 사용자 경험을 개인 서버에서 구현하려는 시도다. 이는 단순히 인터페이스를 흉내 내는 일이 아니다. 모든 데이터를 개인 하드웨어에 저장하고 처리하여, **로컬 우선(local-first)**, **프라이버시 우선(privacy-first)** 원칙을 철저히 지킨다. 외부 서비스에 내 정보를 맡기는 불안감을 해소하고, 데이터가 외부로 유출될 위험을 근본적으로 차단하는 전략이다.

이는 교육 현장에서 중요한 의미를 지닌다. 학생들의 학습 데이터, 개인 정보, 프로젝트 결과물 같은 민감한 자료들은 외부 클라우드 서비스에 저장될 때마다 잠재적 위협에 노출된다. 오디세우스는 이러한 정보 유출의 가능성을 원천 차단하며, 교육 기관이 자신의 데이터를 완전히 통제할 수 있는 길을 제시한다. 그러나 이 모든 것은 단순히 '선언'으로 끝나지 않는다. 사용자가 직접 서버를 관리해야 한다는 조건이 따른다. 이는 곧 기술적 역량의 중요성을 단언한다.

## 교육 현장을 위한 오디세우스의 실질적 기능

오디세우스는 단순히 챗봇 역할만 하는 도구가 아니다. 다양한 기능이 통합된 복합적인 AI 워크스페이스를 지향한다. 다음은 교육적 맥락에서 특히 눈여겨볼 만한 핵심 기능이다.

| 기능 영역 | 주요 내용 | 교육 현장에서의 의미 |
| :-------- | :-------- | :------------------- |
| **챗/에이전트** | 로컬 모델 또는 API 연동 챗, 도구 활용 자동화 | 학생의 맞춤형 학습 도우미, 교사의 반복 업무 자동화 |
| **쿠킹북** | 하드웨어 스캔, 모델 추천 및 설치/제공 | 다양한 오픈소스 LLM을 쉽게 테스트하고 적용 |
| **심층 연구** | 다단계 정보 수집, 분석 및 보고서 생성 | 학생 프로젝트의 탐구 과정 지원, 교사의 자료 조사 효율화 |
| **문서 편집** | AI 지원 마크다운/HTML/CSV 편집기 | 학생의 작문 활동 보조, 교사의 콘텐츠 제작 지원 |
| **기억/기술** | 영구적 기억 및 기술 학습 에이전트 | 학생의 개인화된 학습 경험 강화, 교사의 피드백 자동화 |

챗 기능은 학생들의 질의응답을 돕거나, 특정 주제에 대한 심층 질문을 유도하는 데 활용할 수 있다. 에이전트는 학생에게 과제를 주고 필요한 정보를 스스로 찾아 정리하도록 하거나, 교사의 채점 보조 업무를 자동화하는 등의 가능성을 연다. 쿠킹북은 학교 내 제한된 서버 환경에서도 다양한 오픈소스 LLM을 쉽게 설치하고 운영하며, 모델별 성능을 비교할 수 있도록 돕는다. 이는 특정 기업의 모델에 종속되지 않고 자율적이고 비판적인 모델 선택을 가능하게 한다.

특히 심층 연구 기능은 학생들에게 정보 검색과 분석 과정을 학습시키는 강력한 도구가 된다. 단순히 정보를 요약하는 것을 넘어, 여러 출처를 비교하고 합성하여 시각적 보고서를 만드는 과정은 비판적 사고력을 기르는 데 기여한다. 물론 이는 AI가 전적으로 대신하는 것이 아닌, **인간의 의사결정을 보조하는 역할**로 국한될 때 진정한 교육적 가치를 발휘한다. AI가 연구를 ‘대신’하는 것이 아니라, ‘함께’하는 과정이 중요함이다.

## 기술적 자율성, 그 이면의 현실과 책임

오디세우스는 설치와 운영 방식에 대한 상세한 가이드를 제공한다. 도커(Docker)를 통한 쉬운 배포부터 리눅스, macOS, 윈도우까지 다양한 환경에서의 직접 설치를 지원한다. 특히 엔비디아(NVIDIA)와 AMD 그래픽카드(GPU)를 활용한 모델 가속화 방법에 대한 구체적인 지침도 포함한다. 이는 기술적인 준비가 된 사용자에게는 큰 장점이다.

하지만 동시에 이 지점은 교육 현장에 중요한 질문을 던진다. '얼마나 많은 학교가 이러한 기술적 기반을 갖추고 있는가?' 데이터 주권의 확보는 매우 중요한 목표지만, 이를 위해 기술적 전문성과 관리 부담을 감수해야 한다. `AUTH_ENABLED=true`(인증 활성화) 같은 기본 보안 설정을 지키고, `127.0.0.1`로의 바인딩(내부 네트워크 접근)을 우선하며, 외부 노출 시에는 반드시 HTTPS와 역방향 프록시(reverse proxy)를 통해 보안을 강화해야 한다는 경고는 이러한 책임감을 명확히 한다.

구조적으로 보면, 오디세우스는 강력한 로컬 AI 도구들을 제공하지만, 이는 곧 운영자의 책임 범위 확장을 의미한다. 셸(shell) 접근, 파일 업로드, 모델 다운로드, 웹 검색, 이메일/캘린더 통합, API 토큰 관리 등 민감한 기능들이 모두 로컬 서버에서 이루어지기 때문이다. 이는 단순한 프로그램 설치를 넘어 **서버 관리자의 역할**을 요구하며, 보안 취약점을 만들지 않도록 각별히 유의해야 한다. 학교 정보 교사나 IT 관리자가 없는 환경에서는 이러한 장점들이 실제적인 도전 과제로 다가온다.

## GPU 없는 윈도우에서 오디세우스 시작하기

데이터 주권이라는 명분에 공감해도, 막상 설치 앞에서 멈추는 경우가 많다. 특히 학교의 일반 업무용 PC처럼 외장 그래픽카드(GPU)가 없는 윈도우 환경에서 정말 돌아가는지가 가장 큰 의문이다. 결론부터 적으면, 오디세우스 앱 자체는 GPU 없이도 돌아간다. 공식 문서는 챗·에이전트·기억·문서·이메일·캘린더·심층 연구 같은 핵심 기능이 윈도우에서 완전히 네이티브로 동작한다고 명시한다. 무거운 부분은 앱이 아니라 그 위에서 돌리는 모델 추론이며, 이 추론을 어디서 처리하느냐가 GPU 없는 환경의 핵심 갈림길이다.

설치 전 준비물은 단출하다. 공식 문서 기준 필수 요건은 파이썬 3.11 이상 하나다. 쿠킹북의 백그라운드 모델 다운로드 기능과 에이전트의 셸 도구까지 쓰려면 `bash.exe`를 제공하는 Git for Windows를 함께 설치한다. 로컬에서 모델까지 직접 돌리려면 윈도우에서는 Ollama가 가장 쉬운 경로라고 문서가 직접 권한다.

설치 방식은 크게 도커와 네이티브 직접 설치 두 갈래다. 둘의 성격은 아래와 같다.

| 구분 | 도커(권장) | 네이티브 직접 설치 |
| :--- | :--- | :--- |
| 사전 요구 | Docker 설치 | 파이썬 3.11+ (+ Git for Windows) |
| 부가 서비스 | ChromaDB·SearXNG·ntfy가 함께 묶여 기동 | 직접 구성 |
| 초기 비밀번호 | `docker compose logs odysseus`에 출력 | 실행 터미널에 출력 |
| 접속 주소 | `http://localhost:7000` | `http://localhost:7000` |
| 적합한 경우 | 한 번에 전체 스택을 띄우고 싶을 때 | 파이썬 환경을 직접 다루는 데 익숙할 때 |

네이티브 직접 설치는 한 줄 런처가 가장 간단하다. 가상환경 생성·의존성 설치·초기 설정·서버 기동을 한 번에 처리하며, 다시 실행해도 안전하다.

```powershell
git clone https://github.com/pewdiepie-archdaemon/odysseus.git
cd odysseus
powershell -ExecutionPolicy Bypass -File .\launch-windows.ps1
```

수동으로 단계를 밟고 싶다면 아래와 같다. `python`이 옛 버전을 가리키면 `py -3.12` 등 설치된 3.11 이상 버전을 쓴다.

```powershell
git clone https://github.com/pewdiepie-archdaemon/odysseus.git
cd odysseus
py -3.11 -m venv venv
venv\Scripts\Activate.ps1
pip install -r requirements.txt
python setup.py
python -m uvicorn app:app --host 127.0.0.1 --port 7000
```

도커로 띄울 경우 절차는 더 짧다. 컨테이너가 정상 상태가 되면 같은 주소로 접속한다.

```bash
git clone https://github.com/pewdiepie-archdaemon/odysseus.git
cd odysseus
cp .env.example .env
docker compose up -d --build
```

어느 방식이든 첫 기동 시 `admin` 관리자 계정이 만들어지고 임시 비밀번호가 출력된다. 네이티브 설치는 실행 터미널에, 도커는 `docker compose logs odysseus` 로그에 같은 줄이 찍힌다. 이 값으로 처음 로그인한 뒤 설정에서 바꾼다. `http://localhost:7000`으로 접속하면 된다.

이제 모델 연결이다. 윈도우에서 로컬 모델을 돌리는 가장 쉬운 길은 Ollama이며, 이는 문서가 직접 권하는 방식이다. Ollama를 설치한 뒤 오디세우스 설정에서 엔드포인트를 추가한다.

```text
http://localhost:11434/v1
```

도커로 오디세우스를 띄웠고 Ollama는 호스트(윈도우)에서 돌고 있다면, 컨테이너 안에서 호스트를 가리키는 주소를 쓰고 Ollama가 자기 루프백 밖에서도 듣도록 띄워야 한다.

```bash
OLLAMA_HOST=0.0.0.0:11434 ollama serve
```

이 경우 설정에 추가할 엔드포인트는 `http://host.docker.internal:11434/v1`이다. 이미 윈도우에 Ollama를 설치해 둔 사용자는 대개 설정에서 이 엔드포인트만 추가하면 된다.

GPU가 없는 환경에서 반드시 짚어야 할 현실이 있다. 공식 문서는 vLLM·SGLang 같은 엔진으로 모델을 로컬에서 가속 제공하려면 리눅스나 WSL2가 필요하다고 분명히 한다. 즉 외장 GPU 없는 일반 윈도우 PC에서 큰 모델을 빠르게 돌리는 경로는 문서상 존재하지 않는다. 그렇다고 길이 막힌 것은 아니다. 문서는 모델 추론이 무거운 부분이고 작은 호스트는 외부 API나 원격 모델 서버에 연결하면 된다고 안내하며, 챗은 OpenAI·OpenRouter·GitHub Copilot 같은 외부 제공자와도 곧바로 연결된다.

선택지는 다음과 같이 정리된다.

| 모델 처리 위치 | GPU 필요 | 데이터 주권 | 속도와 현실 |
| :--- | :--- | :--- | :--- |
| CPU만으로 로컬 추론(Ollama) | 불필요 | 완전 로컬 | 모델이 클수록 느려짐. 작은 모델 위주로 |
| GPU 로컬 가속(vLLM/llama.cpp) | 필요(리눅스/WSL2) | 완전 로컬 | 빠름. 일반 윈도우 업무 PC에는 부적합 |
| 외부 API 연동(OpenAI 등) | 불필요 | 외부로 전송됨 | 빠름. 단 데이터가 외부로 나감 |

여기서 정직하게 밝힐 부분이 있다. 공식 문서는 CPU 전용으로 어떤 소형 모델을 몇 GB 메모리에서 초당 몇 토큰으로 돌릴 수 있는지 같은 구체적 수치를 제시하지 않는다. 다만 모델이 무거울수록 추론이 느려진다는 점, 그래서 작은 호스트는 외부 API나 원격 서버 연결이 권장된다는 점은 문서에 근거한 사실이다. GPU 없는 윈도우 사용자라면 처음에는 작은 모델로 응답 속도를 직접 체감해 보고, 실시간성이 필요한 수업 활동에는 외부 API 연동을, 민감한 데이터를 다루는 작업에는 느리더라도 로컬 추론을 쓰는 식으로 용도를 나누는 편이 현실적이다. 데이터 주권과 속도 사이의 선택은 결국 무엇을 더 양보할 수 없는지에 달려 있다.


## 교육 현장의 '어떻게'에 대한 제언

우리는 클라우드 AI의 편리함이 가져오는 잠재적 위험을 인지하고, 오디세우스와 같은 로컬 우선 솔루션이 제공하는 데이터 주권과 프라이버시의 가치를 인정한다. 이제 중요한 것은 '어떻게' 이 가치를 우리 현장에 가져올 것인가 하는 문제다.

1.  **점진적 도입과 실험**: 처음부터 완벽한 환경을 구축하려 하기보다, 소규모 실험을 권장한다. 예를 들어, 한 학년의 교사 한두 명이 개인 컴퓨터에 Ollama를 설치하고, 그 위에 오디세우스를 연동하여 모델의 성능과 실제 교육 적용 가능성을 탐색하는 방식이다. 이 과정에서 발생하는 문제점과 해결책을 구체적으로 문서화하고 공유하는 것이 중요하다. 동학년 점심시간 대화에서 "어제 오디세우스에 특정 모델을 올려봤더니 이런 오류가 나던데, 이렇게 해결했어!"와 같은 한두 문장 공유가 시작이다.
2.  **교내 IT 전문 인력과의 협력**: 학교에 IT 관리자나 전산에 능숙한 교사가 있다면, 그들과 함께 오디세우스의 서버 구축을 논의해야 한다. 데이터 보안과 시스템 안정성을 확보하려면 전문적인 지식이 필수적이다. 이들이 시스템을 구축하고, 교사들은 활용 방안에 집중하는 역할 분담이 가장 현실적인 접근이다.
3.  **윤리적 활용 가이드라인 마련**: 강력한 로컬 AI 도구들은 학생의 표절이나 과도한 의존으로 이어질 수 있다. 오디세우스의 '심층 연구'나 '에이전트' 기능은 학생들의 학습을 돕는 보조 도구로 명확히 정의하고, AI가 생성한 결과물을 비판적으로 검토하고 재구성하는 과정을 반드시 포함하도록 지도해야 한다. AI의 역할이 '대신'이 아닌 '협력'에 있음을 분명히 하는 교육 철학이 전제될 때만, 이러한 도구는 긍정적인 영향을 미친다.

오디세우스는 AI 시대를 맞아 데이터 주권을 확보하고 교육적 자율성을 높이려는 우리에게 매력적인 대안임에 분명하다. 그러나 이 항해는 단순히 기술의 도입을 넘어, **현장의 인식과 문화, 그리고 기술적 역량의 성장**이라는 더 큰 파도를 헤쳐나가야 함을 단언한다. 당신의 학교는 AI 시대의 데이터 주권을 어떻게 확보할 계획인가?
